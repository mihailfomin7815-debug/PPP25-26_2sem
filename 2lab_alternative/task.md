# Задание 2. Веб‑сервис над данными ETL (FastAPI + БД)

## Исходные данные

В качестве источника данных используются результаты **ETL‑процесса** из альтернативного задания (агрегированные и унифицированные данные из нескольких внешних сайтов, загруженные в единую БД).

Рекомендуется **не изменять схему и содержимое исходной ETL‑БД**, а поверх неё строить веб‑сервис (можно добавить служебные таблицы, но базовые сущности ETL лучше оставить неизменными).

---

## Цель задания

Разработать веб‑API на **FastAPI** с использованием **реляционной БД** (PostgreSQL/SQLite и т.п.), которое:

- использует данные, полученные из ETL‑процесса;
- имеет **как минимум 3 связанные таблицы** (связи `1‑ко‑многим` или `многие‑ко‑многим`);
- предоставляет **не менее 10 HTTP‑ручек**, включая:
  - `GET`
  - `POST`
  - `PUT`
  - `PATCH`
  - `DELETE`.

---

## Базовые требования

1. **Модель данных**
   - Описать ORM‑модели (например, через SQLAlchemy) для:
     - не менее **3 логически связанных таблиц**;
     - обязательно должны присутствовать **внешние ключи** и связи:
       - пример: `Source` → `Item` → `ItemEvent` (источник, сущность, история изменений/лог событий).
   - Таблицы должны базироваться на данных ETL (например, `sources`, `items`, `prices`, `categories` и т.д.).

2. **API‑эндпоинты (не менее 10)**
   - Обязательно должны присутствовать:
     - как минимум один `GET` список (`GET /items`, `GET /sources`, ...);
     - как минимум один `GET` по ID (`GET /items/{id}`);
     - как минимум один `POST` (создание сущности);
     - как минимум один `PUT` (полная замена ресурса);
     - как минимум один `PATCH` (частичное обновление);
     - как минимум один `DELETE` (удаление / деактивация).
   - Остальные ручки — по выбору: фильтрация, поиск, агрегаты, получение связанных данных и т.п.
   - Рекомендуется:
     - использовать **Pydantic‑схемы** для валидации входных/выходных данных;
     - реализовать **обработку ошибок** (404, 400, 422 и т.д.).

3. **Архитектура**
   - Проект должен быть структурирован, пример:
     - `app/main.py` (инициализация FastAPI);
     - `app/models.py` / `app/db.py` (конфигурация БД и модели);
     - `app/schemas.py` (Pydantic‑схемы);
     - `app/routes/*.py` (роутеры);
     - `app/services/*.py` (бизнес‑логика) — по желанию.
   - Поддержка **миграций** (Alembic или аналог) приветствуется, но не обязательна.

## Дополнительные задания

### 1. Поддержка асинхронных долгих задач (Celery + Redis)

**Задача:** интегрировать в проект очередь задач для длительных операций (пересчёт статистики, повторный импорт, отчёты и т.п.) с использованием **FastAPI + Celery + Redis**.  
Ожидаемое поведение со стороны клиента:

- Клиент отправляет запрос на запуск долгой задачи (`POST /tasks/...`).
- API **не ждёт завершения задачи**, а:
  - ставит задачу в очередь Celery;
  - возвращает клиенту **идентификатор задачи** (`task_id`) и, например, начальный статус `pending`.
- Клиент может:
  - периодически опрашивать эндпоинт статуса (`GET /tasks/{task_id}`);
  - или (в связке с WebSocket) получать обновления статуса в реальном времени.

**Общее описание связки:**

- **FastAPI (producer)**: принимает HTTP‑запрос, формирует задачу `celery_app.send_task(...)` или `@celery_app.task(...).delay(...)`, кладёт её в Redis и сразу отдаёт ответ. 
- **Redis (broker / backend)**: хранит очередь задач и состояние (по желанию — используется и как результат‑backend). 
- **Celery worker (consumer)**: отдельный процесс, который:
  - читает задачи из Redis;
  - выполняет тяжёлую работу;
  - обновляет статус/результат задачи.

Минимально требуется:

- настроить Celery‑приложение (например, `celery_app = Celery(..., broker="redis://...", backend="redis://...")`);
- сделать хотя бы одну реальную долгую задачу (условно — `rebuild_stats` или `reimport_source`);
- добавить ручки:
  - `POST /tasks/rebuild_stats` — запуск задачи, возврат `task_id`;
  - `GET /tasks/{task_id}` — получение статуса/результата.

### 2. Взаимодействие по WebSocket

**Вариант A: отслеживание прогресса задач**

- Реализовать WebSocket‑эндпоинт, например `GET /ws/{client_id}`.
- Клиент:
  - устанавливает WebSocket‑соединение;
  - запускает долгую задачу (см. задание с Celery);
  - получает по WebSocket обновления статуса (`queued`, `running`, `% прогресса`, `finished`, `failed`). 
- Сервер:
  - хранит соответствие `client_id` ↔ WebSocket‑соединения;
  - при обновлении статуса в Celery (или через отдельный воркер) отправляет сообщение клиенту по WebSocket.

**Вариант B: кооперация клиентов**

- Один клиент создаёт задачу (например, задачу на модерацию/проверку данных).
- Другой клиент:
  - получает список задач для обработки (`GET /tasks/for_review`);
  - берёт задачу в работу (`POST /tasks/{id}/take`);
  - по WebSocket все подключённые клиенты получают обновление состояния задачи (взята в работу, завершена, отклонена и т.п.).

Для зачёта достаточно реализовать **один из вариантов**, но желательно продумать формат сообщений (`{"task_id": "...", "status": "...", "progress": ...}` или похожий).

### 3. Юнит‑тесты без запуска Uvicorn

Необходимо реализовать набор **юнит/интеграционных тестов** для API, которые:

- запускаются через `pytest` (или `unittest`);
- используют `fastapi.testclient.TestClient` для обращения к приложению **без запуска uvicorn**. 
- покрывают:
  - базовые CRUD‑операции (как минимум по одной ручке каждого типа: GET/POST/PUT/PATCH/DELETE);
  - хотя бы один happy‑path для долгой задачи (например, проверка, что `POST /tasks/...` возвращает `task_id` корректного формата);
  - хотя бы один тест на валидацию/ошибки (например, 404 или 422).

Примерный паттерн:

```python
from fastapi.testclient import TestClient
from app.main import app

client = TestClient(app)

def test_get_items():
    response = client.get("/items")
    assert response.status_code == 200
```
